# 用户管理与权限优化实施计划（RBAC + 行级权限）

## 1. 目标与范围
- 引入基于角色的访问控制（RBAC），角色初步为 `User` 与 `Admin`。
- 后端实现会话相关数据的行级权限控制（RLS 逻辑）：`User` 仅可 CRUD 自己关联的私有对话，禁止越权。
- `Admin` 放开 `user_id` 限制，可获取全局对话列表并执行管理操作。
- 本次不做数据迁移，仅覆盖当前未上线环境。

## 2. 现状基线（代码落点）
- 认证与角色校验中间件：`src/server.ts` 中 `authenticateToken`、`requireAdmin`。
- 用户模型与角色字段：`src/db.ts` 的 `users` 表与 `User.role`。
- 会话存储接口：`src/db.ts` 的 `getSession/setSession/deleteSession/getAllSessions`。
- 聊天/会话接口入口：`src/server.ts` 的 `/api/chats`、`/api/groups/:jid/messages`、`/api/groups/:jid/stream`。

## 3. 实施步骤

### 步骤 A：统一 RBAC 类型与鉴权上下文
1. 在后端定义并统一角色枚举/字面量类型（`User`、`Admin`），避免各处硬编码字符串。
2. 扩展鉴权上下文（`AuthenticatedRequest.user`）的类型约束，明确包含 `id` 与 `role`。
3. 保持 `requireAdmin` 作为“角色门禁”，并新增可复用的角色判断辅助函数（如 `isAdmin(user)`）供业务层调用。

### 步骤 B：建立会话与用户的归属关系（RLS 依据）
1. 在会话表结构中增加 `user_id` 字段（当前环境允许直接调整 schema，不做迁移逻辑）。
2. 为 `sessions.user_id` 建立索引，提高按用户过滤查询性能。
3. 在会话写入路径中强制写入归属用户：创建/更新会话时记录 `user_id`。
4. 新增会话数据访问函数：
   - `getSessionsByUserId(userId)`：返回指定用户会话。
   - `getAllSessions()`：供 `Admin` 全量查询。
   - `getSessionByIdForUser(sessionId, user)`：按角色自动执行所有权过滤。

### 步骤 C：在会话 CRUD 路径落地行级权限控制
1. 列表读取（Read-List）：  
   - `User` 仅返回 `user_id = 当前用户.id` 的会话。  
   - `Admin` 返回全量会话。
2. 单会话读取（Read-One）：  
   - `User` 仅可访问自身会话。  
   - 非法访问返回 403/404（按现有 API 风格统一）。
3. 更新（Update）：  
   - `User` 仅可修改自身会话。  
   - `Admin` 可修改任意会话。
4. 删除（Delete）：  
   - `User` 仅可删除自身会话。  
   - `Admin` 可删除任意会话。
5. 在所有会话相关 API/处理器中统一复用权限判断逻辑，避免分散 if-else 导致遗漏。

### 步骤 D：管理员全局查询能力
1. 在会话列表 API 中增加管理员分支，支持全局会话查询（不受 `user_id` 限制）。
2. 如接口支持筛选参数，允许 `Admin` 使用 `user_id` 进行条件过滤；`User` 传入该参数时强制忽略或拒绝。
3. 保持响应结构兼容前端当前数据消费方式，避免无关字段破坏页面。

### 步骤 E：用户管理接口与前端联动（最小必要改动）
1. 后端保持 `/api/users` 系列接口继续仅 `Admin` 可用，与 RBAC 规则对齐。
2. 前端继续以令牌中的角色决定页面展示；确保普通用户不出现管理员入口（若已存在则复核，不重复改造）。
3. 若会话列表前端依赖管理员全量数据，补齐请求参数与空态处理，保持与后端新权限语义一致。

### 步骤 F：安全与错误处理一致性
1. 所有越权访问统一返回明确错误码与消息，避免泄露目标资源存在性细节。
2. 审查日志内容，避免输出敏感令牌或用户隐私数据。
3. 保留现有 JWT 认证流程，不改变登录/发 token 主链路。

### 步骤 G：验证与回归
1. 编写/补充后端权限测试（或最小集成测试）覆盖：
   - `User` 访问自己的会话：允许。
   - `User` 访问他人会话：拒绝。
   - `Admin` 访问任意会话：允许。
   - 会话创建后自动绑定 `user_id`。
2. 执行现有测试与关键接口回归，确保未引入认证回归问题。
3. 对关键接口进行手工验证：登录后用不同角色 token 调用同一路径，确认返回差异符合预期。

## 4. 交付清单
- RBAC 类型与权限辅助函数落地。
- 会话表新增 `user_id` 并在读写路径生效。
- 会话 CRUD 全链路 RLS 逻辑生效。
- Admin 全局会话查询能力可用。
- 权限测试与回归验证通过。

## 5. 非目标与约束
- 非目标：历史数据迁移、灰度发布策略、多角色体系扩展（如审计员等）。
- 约束：在当前“未上线、无迁移负担”前提下实施，优先保证权限正确性与代码可维护性。
