# 前端页面比例与 UI 基线优化设计

**日期：** 2026-04-13

**目标：** 让系统在 `1920x1080` 分辨率、浏览器 `100%` 缩放下获得接近当前 `75%` 缩放时的观感，同时保持现有后台工具的紧凑信息密度，不做视觉语言重构。

**范围：**
- `frontend/src/components/Layout.tsx`
- `frontend/src/components/Sidebar.tsx`
- `frontend/src/pages/Dashboard.tsx`
- `frontend/src/pages/Chat.tsx`
- `frontend/src/pages/KnowledgeBase.tsx`
- `frontend/src/pages/SkillsAdmin.tsx`
- `frontend/src/pages/SkillsCatalog.tsx`
- `frontend/src/pages/Users.tsx`
- `frontend/src/pages/Login.tsx`
- 与上述页面直接相关的通用样式文件和局部组件

**非目标：**
- 不修改路由、权限、数据流和后端 API
- 不改整站配色体系，只做必要的层级与对比度微调
- 不把页面改造成大留白、展示型官网风格
- 不通过浏览器缩放、CSS `zoom` 或整体 `transform: scale()` 取巧

## 1. 背景与问题

当前前端页面在默认 `100%` 缩放下存在以下问题：

- 首页与部分管理页大量使用 `10px` 到 `11px` 文字，阅读基线偏小
- 侧栏宽度、按钮高度、卡片内边距偏紧，导致整体比例更像“被压缩过”的后台
- 不同页面各自维护尺寸，缺少统一的后台尺寸基线
- 用户当前依赖浏览器 `75%` 缩放获得更合适的观感，说明设计比例本身失衡

同时，聊天页存在额外约束：

- 当前模型对话消息正文已经偏大，不能随着全局优化再继续放大
- 聊天页需要维持工作台式密度，避免消息区变得臃肿

## 2. 需求结论

经过确认，本次设计约束如下：

- 目标使用环境为 `1920x1080`
- 浏览器缩放目标为 `100%`
- 保持当前信息密度，不做明显“舒展化”改版
- 全站需要统一调整，而不是只修复单页
- 聊天页是例外：消息正文不跟随全局统一放大

## 3. 方案对比

### 3.1 方案 A：直接放大全局基础字号

通过提升 `html` 或 `body` 基础字号，让所有页面整体变大。

优点：

- 改动量最小
- 见效快

缺点：

- 会把消息正文、表格、弹窗等一起放大
- 现有页面中不同元素原本比例不一致，整体放大会放大失衡
- 不能精确处理聊天页例外约束

### 3.2 方案 B：建立后台尺寸基线，再逐页收口

先定义统一的后台尺寸规则，再将各页面对齐到同一基线；对聊天页单独设置消息区例外。

优点：

- 最符合“全站统一调整但保持密度”的目标
- 能同时修复文字、容器、控件和表格比例
- 可以保留当前后台风格，不必重做视觉语言
- 能精确处理聊天页的特殊限制

缺点：

- 改动量高于方案 A
- 需要对多个页面做定向收口

### 3.3 方案 C：逐页人工微调

不建立全局规则，只在页面内局部放大或缩小。

优点：

- 初期灵活

缺点：

- 维护性差
- 页面间容易继续失衡
- 后续迭代仍会回到同样问题

### 3.4 推荐方案

采用 **方案 B：建立后台尺寸基线，再逐页收口**。

原因：

- 这是唯一同时满足“全站统一”“保持密度”“聊天页例外”三项约束的方案
- 适合当前项目已经存在多个管理页和工作台页面的现状
- 可以在不改业务逻辑的前提下完成一次可持续的 UI 比例修正

## 4. 设计原则

### 4.1 统一后台基线

后台页面统一遵循同一套字体、间距、容器和控件高度规则，不允许每个页面继续自由使用过小字号。

### 4.2 修正比例，不追求显著放大

目标是恢复在 `100%` 缩放下的自然观感，而不是做“更大更松”的视觉升级。

### 4.3 聊天页特判

聊天页继续使用独立的阅读尺度。会话列表、输入区和辅助控件可以调整，但消息正文保持中性尺寸，不随管理页一起放大。

### 4.4 优先保持现有结构

不推翻现有页面布局；重点修正标题层级、留白、控件尺寸、表格密度和说明文字比例。

## 5. 全局尺寸基线

### 5.1 字号阶梯

建议统一为以下后台字号区间：

- 页面主标题：`24px` 左右
- 区块标题：`18px` 到 `20px`
- 卡片标题、表格主要标题：`14px` 到 `16px`
- 正文基线：`14px`
- 次级说明：`12px` 到 `13px`
- 标签、状态、辅助字段：`12px`

规则：

- 避免在核心页面大面积使用 `10px`
- `11px` 仅允许在极少量密集型辅助字段中出现
- 正文与说明文字必须形成清晰层级，不能同时都偏小

### 5.2 间距与容器

建议统一以下节奏：

- 页面外层水平内边距使用一致阶梯，桌面端不再出现过窄内容贴边
- 卡片内边距统一上调一小档
- 区块之间的 `gap` 与列表行间距使用固定档位，避免同页多套节奏
- 表格行高略增，保证可扫读性，但不牺牲信息密度

### 5.3 控件高度体系

建议建立三档控件高度：

- 紧凑控件：用于表格工具条、次级筛选器
- 标准控件：用于页面主按钮、输入框、导航项
- 强调控件：用于首页主 CTA 或关键操作

统一要求：

- 按钮、输入框、侧栏导航项、弹窗操作按钮使用同一高度语言
- 不允许出现文字放大后控件仍显得扁薄的情况

### 5.4 侧栏与主框架

建议调整侧栏宽度和头部区比例，使其与正文区建立更稳定的左右关系：

- 侧栏比当前略宽一档
- Logo 区、导航项、用户区使用统一高度与字体层级
- 主内容区在宽屏下保留稳定的可读宽度，不依赖缩放压缩页面

## 6. 页面级收口方案

### 6.1 Dashboard

首页是本次问题最明显的页面，优先级最高。

调整重点：

- 上调主标题、说明文字与顶部统计胶囊的层级
- 统一卡片标题、描述、数值和图标容器比例
- 将任务表格表头、单元格、状态标签提升到可读但仍紧凑的区间
- 保留当前的卡片网格和分区结构，不重做布局

预期结果：

- 默认 `100%` 下首页不再显得过小
- 卡片数值与说明不再挤在同一视觉层
- 首页仍然是高信息密度控制台，而非展示页

### 6.2 Chat

聊天页必须与其他管理页分开处理。

调整重点：

- 会话列表、侧边抽屉、输入区、工具卡片按照全站基线轻微校正
- 消息时间、状态、辅助说明维持次级层级
- 消息正文保持中性阅读尺寸，不跟随全局放大
- 通过调整行高、气泡内边距和局部间距来提升阅读稳定性，而不是直接加大消息字体

明确限制：

- 不允许把模型对话消息正文做明显放大
- 不允许因为统一尺寸而削弱聊天工作台的信息密度

### 6.3 KnowledgeBase

知识库页同时承载目录、搜索、正文阅读和编辑，是典型的工作区页面。

调整重点：

- 统一目录树行高和选中态尺寸
- 搜索框、工具按钮、操作区高度与全局控件体系对齐
- 正文预览与编辑区使用稳定的正文基线和段落间距
- 侧栏、内容区与工具条的内边距节奏统一

预期结果：

- 看起来更像专业文档工作区
- 保持紧凑，但不再依赖浏览器缩放提高可读性

### 6.4 SkillsAdmin / SkillsCatalog

这两页以列表、卡片、树结构和编辑抽屉为主。

调整重点：

- 技能卡片标题、摘要、更新时间恢复合理比例
- 文件树节点、面板标题、编辑器工具条与按钮高度统一
- 抽屉标题、说明、操作按钮与内容区正文建立稳定层级

目标：

- 技能管理页与知识库页共享同一套后台工作区比例

### 6.5 Users

用户管理页主要是表格与弹窗场景。

调整重点：

- 表头、单元格、标签和操作按钮统一尺寸
- 新增按钮、错误提示、表单输入框与弹窗标题对齐全局基线
- 保留 CRUD 流程和页面结构

### 6.6 Login

登录页只做比例校正，不做风格改版。

调整重点：

- 标题、说明、输入框和按钮与后台主站尺寸基线统一
- 保持简洁专业风格，避免形成独立视觉系统

## 7. 实施边界

### 7.1 允许的实现方式

- 提取少量全局样式变量、通用类名或常量类组合
- 统一页面中的字号、间距、容器宽度和控件高度
- 对聊天页局部样式做例外处理

### 7.2 不建议的实现方式

- 使用全局 `zoom`
- 使用大面积的 `text-[10px]`、`text-[11px]`
- 为每个页面单独发明一套新的尺寸体系
- 为了“看起来更大”而无差别增加所有元素尺寸

## 8. 验证标准

### 8.1 视觉验证

在 `1920x1080`、浏览器 `100%` 缩放下，应满足：

- 首页主内容不再显得细小发虚
- 管理页表格和表单具备稳定可读性
- 知识库与技能页的目录、正文、工具条层级明确
- 聊天消息正文不显臃肿，输入区与侧边栏比例更协调

### 8.2 响应式验证

至少验证以下宽度不出现明显退化：

- `375px`
- `768px`
- `1024px`
- `1440px`
- `1920px`

### 8.3 回归验证

- 不引入新的水平滚动问题
- 不遮挡现有固定区域与抽屉
- 不破坏深色与浅色主题切换
- 不影响现有交互行为和功能流程

## 9. 实施顺序建议

建议按以下顺序推进：

1. 统一全局基线：字体、侧栏、控件、高度和间距
2. 优先收口 `Dashboard`
3. 处理 `KnowledgeBase` 与 `Skills` 系列页面
4. 处理 `Users` 与 `Login`
5. 最后单独校正 `Chat`，确保消息正文不会被全局基线误伤

这样可以先建立统一规则，再用聊天页作为特例做最终收口，减少返工。
