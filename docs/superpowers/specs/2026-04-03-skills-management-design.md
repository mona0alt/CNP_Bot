# Skills 管理与会话绑定设计

**日期：** 2026-04-03

**目标：** 为项目增加基于文件系统的全局 skills 库管理能力，并支持普通用户为自己的 Web 会话选择需要生效的 skills。

**范围：**
- 管理员管理全局 skills 库
- 普通用户只读查看全局 skills 库
- 普通用户为自己的 `web:*` 会话选择 skills
- 会话运行中立即同步，未运行时延后到下次启动生效

**非目标：**
- 本期不支持已注册群组会话绑定 skills
- 本期不引入 skill 版本管理、审核流、标签系统
- 本期不实现用户侧完整文件树浏览

## 1. 背景与现状

当前项目已有以下基础能力：

- 后端基于 Express，已经具备 JWT 鉴权与 `admin` / `user` 角色隔离
- 前端已有管理页模式，现有 `/users` 页面可作为管理员 CRUD 页的实现参考
- 系统已有 skills 相关能力与目录约定，但没有面向 Web UI 的全局 skills 管理与会话绑定功能
- Web 聊天会话以 `web:*` 标识，且已具备会话创建、消息发送、运行态判断等能力

本次设计新增的是一套 Web 管理能力，而不是重做底层 skill 机制。

## 2. 需求结论

经过确认，需求约束如下：

- 管理员维护“全局 skills 库”
- 全局库中的每个 skill 至少包含 `SKILL.md`
- 管理员可在前端进行完整文件管理：
  - 新建文件
  - 新建目录
  - 编辑文件内容并保存
  - 删除文件或目录
  - 重命名或移动文件、目录
  - 上传 zip 导入 skill
- zip 上传后由后端自动解压并写入全局 skills 存储目录
- 若 zip 导入出的 skill 名称与现有 skill 同名，则拒绝上传，不覆盖
- 普通用户只能查看全局 skills 库，不能修改
- 普通用户可以为自己的 Web 会话选择需要生效的 skills
- 会话创建时可选，创建后也允许修改
- 若会话当前运行中，保存后立即同步到会话生效目录，对后续消息生效
- 若会话未运行，仅保存绑定关系，下次启动时自动同步

## 3. 方案对比

### 3.1 方案 A：纯文件系统方案

以磁盘目录作为全局 skills 库的真实来源，数据库只存会话与 skill 的绑定关系。

优点：

- 贴合“像文件管理器一样管理 skill 目录”的需求
- 读写链路清晰，易于调试
- 不需要维护数据库与真实文件的双向一致性

缺点：

- 需要单独处理路径安全、并发写保护与 zip 校验
- 复杂查询能力有限

### 3.2 方案 B：文件系统 + 元数据索引

真实文件仍保存在磁盘，但额外引入 `skills` 与 `skill_files` 元数据表。

优点：

- 便于后续增加搜索、标签、版本等能力

缺点：

- 本期明显过度设计
- 数据库与磁盘内容容易漂移

### 3.3 方案 C：Git 仓库化管理

将全局 skills 库视作一个独立仓库，上传与编辑都形成提交。

优点：

- 天然具备版本历史和回滚能力

缺点：

- 明显超出当前需求
- 交互与实现复杂度过高

### 3.4 推荐方案

采用 **方案 A：纯文件系统方案**。

原因：

- 满足本期全部功能边界
- 与现有项目结构和权限体系最容易对接
- 可以在未来平滑演进为“文件系统 + 元数据索引”的方案

## 4. 总体架构

建议将新增能力拆为四个边界明确的模块：

### 4.1 Global Skills Store

负责全局 skills 库的目录与文件管理：

- 列举 skill 列表
- 返回目录树
- 读取文件内容
- 保存文件内容
- 新建文件与目录
- 删除、重命名、移动
- zip 解压导入

### 4.2 Session Skill Binding

负责 Web 会话与已选 skills 的关系管理：

- 读取某个会话当前绑定的 skills
- 保存某个会话当前绑定的 skills
- 校验用户只能操作自己有权限访问的会话

### 4.3 Session Skill Sync

负责把全局 skills 同步到某个会话的生效目录：

- 会话运行中保存后立即同步
- 会话未运行时只记录绑定关系
- 会话启动时自动完成延迟同步
- 记录最近同步状态

### 4.4 Skills UI

负责两类前端界面：

- 管理员：全局 skills 文件管理页
- 用户：全局 skills 只读目录与会话绑定入口

## 5. 存储布局

建议采用以下目录结构：

```text
data/
  skills/
    global/
      <skillName>/
        SKILL.md
        ...
    sessions/
      <chatJid>/
        active/
          <skillName>/
            SKILL.md
            ...
```

### 5.1 全局库

- 根目录：`data/skills/global/`
- 每个 skill 占用一个一级子目录：`data/skills/global/<skillName>/`
- skill 目录必须包含 `SKILL.md`

### 5.2 会话生效目录

- 根目录：`data/skills/sessions/<chatJid>/active/`
- 当前会话每个已启用 skill 同步为一个子目录

### 5.3 设计原则

- 全局库和会话生效目录严格分离
- 管理员修改全局库时，不直接改动运行中的会话目录
- 会话目录代表“当前会话已生效的技能快照”

## 6. 数据模型

本期数据库只保存绑定关系与同步状态，不保存全局文件树元数据。

### 6.1 `session_skill_bindings`

建议新增表：

- `chat_jid TEXT NOT NULL`
- `skill_name TEXT NOT NULL`
- `created_at TEXT NOT NULL`
- `updated_at TEXT NOT NULL`

约束：

- 主键或唯一索引：`(chat_jid, skill_name)`

用途：

- 记录某个 Web 会话当前绑定了哪些全局 skill

### 6.2 `session_skill_sync_state`

建议新增表：

- `chat_jid TEXT PRIMARY KEY`
- `status TEXT NOT NULL`
- `last_synced_at TEXT`
- `error_message TEXT`
- `updated_at TEXT NOT NULL`

状态建议：

- `pending`
- `synced`
- `failed`

用途：

- 为前端展示“已保存待生效 / 已同步 / 同步失败”提供依据

### 6.3 生命周期清理规则

- 删除 Web 会话时，应同时清理：
  - `session_skill_bindings` 中对应记录
  - `session_skill_sync_state` 中对应记录
  - `data/skills/sessions/<chatJid>/` 会话技能目录

### 6.4 不新增全局 skills 元数据表

原因：

- 全局库天然存在于文件系统
- 直接扫描目录可避免数据库与真实文件状态不一致
- 本期未要求标签、搜索、版本等需要索引的数据能力

## 7. 核心规则

### 7.1 skill 目录规则

- 一个 skill 对应一个顶级目录
- skill 目录中必须存在 `SKILL.md`
- `skillName` 必须是安全目录名，不允许：
  - 路径分隔符
  - `..`
  - 空名称
  - 隐藏或保留特殊名称

### 7.2 文件编辑规则

- 仅文本文件支持在线编辑与保存
- 二进制文件仅展示基本信息，不支持在线编辑
- 删除目录默认递归删除，但前端必须二次确认
- 重命名和移动后的目标路径仍必须位于全局 skills 根目录下
- 对顶层 skill 目录的重命名或删除，必须触发绑定关系联动处理

### 7.3 顶层 skill 目录联动规则

当管理员操作 `data/skills/global/<skillName>/` 这一层级时，按“skill 实体变更”处理：

- 顶层 skill 目录重命名：
  - 视为 skill 名称变更
  - 更新所有受影响的 `session_skill_bindings.skill_name`
  - 对受影响会话标记为 `pending`
  - 若会话正在运行，则立即触发重同步
- 顶层 skill 目录删除：
  - 删除成功后，移除所有受影响的会话绑定
  - 对受影响会话标记为 `pending`
  - 若会话正在运行，则立即触发重同步以移除该 skill

这样可以保持“全局库名称”与“会话绑定名称”一致，避免顶层目录被改名后产生悬挂绑定。

### 7.4 zip 导入规则

- 只接受单个 zip 文件
- 后端先解压到临时目录
- 只有当解压结果满足以下条件时才允许导入：
  - 顶层可识别出单个 skill 目录
  - skill 目录下存在 `SKILL.md`
- 若 skill 名称已存在于全局库中，则导入失败
- 校验失败时，整个导入过程回滚，不留下半成品目录

## 8. 后端 API 设计

### 8.1 管理员接口

以下接口全部要求：

- 已登录
- 角色为 `admin`

建议接口如下：

#### `GET /api/skills`

用途：

- 返回全局 skills 列表

返回字段建议包含：

- `name`
- `has_skill_md`
- `updated_at`

#### `GET /api/skills/tree`

用途：

- 返回全局库目录树

参数：

- 可选 `skill=<name>`，用于只查看单个 skill 目录树

#### `GET /api/skills/file?path=...`

用途：

- 读取指定文本文件内容

#### `PUT /api/skills/file`

用途：

- 保存指定文本文件内容

请求体建议包含：

- `path`
- `content`

#### `POST /api/skills/fs`

用途：

- 新建文件或目录

请求体建议包含：

- `parentPath`
- `name`
- `type`，值为 `file` 或 `directory`

#### `PATCH /api/skills/fs`

用途：

- 重命名或移动文件、目录

请求体建议包含：

- `fromPath`
- `toPath`

#### `DELETE /api/skills/fs?path=...`

用途：

- 删除文件或目录

#### `POST /api/skills/upload-zip`

用途：

- 上传 zip 并导入 skill

行为：

- 先解压到临时目录
- 做结构与命名校验
- 校验通过后原子移动到全局库

#### 顶层目录联动要求

当管理员通过文件管理接口重命名或删除顶层 skill 目录时，接口行为必须同时完成：

- 绑定关系更新或清理
- 受影响会话同步状态更新
- 运行中会话的即时重同步触发

如果联动步骤失败，则整个顶层目录操作应视为失败，避免产生半完成状态

### 8.2 普通用户可访问接口

以下接口要求：

- 已登录
- 具备对应会话访问权限

#### `GET /api/skills/catalog`

用途：

- 返回全局 skills 列表，供普通用户只读浏览

#### `GET /api/chats/:jid/skills`

用途：

- 返回当前会话已绑定的 skills 和同步状态

返回内容建议同时包含：

- `selectedSkills`
- `syncStatus`
- `lastSyncedAt`
- `errorMessage`

#### `PUT /api/chats/:jid/skills`

用途：

- 整体替换某个会话当前绑定的 skills 列表

行为：

- 校验 skill 名称都存在于全局库
- 保存绑定关系
- 若会话运行中，立即触发同步
- 若会话未运行，只写入绑定关系并标记为 `pending`

#### `POST /api/chats`

现有创建 Web 会话接口建议扩展支持初始 skills 选择。

新增请求字段建议：

- `skills?: string[]`

行为：

- 创建会话时同时保存初始绑定关系
- 若会话创建后立即进入运行态，可直接触发首次同步
- 若未运行，则写入绑定关系并标记为 `pending`

#### `POST /api/chats/:jid/skills/sync`

用途：

- 手动触发一次会话 skills 重同步

说明：

- 该接口不是主流程必须项
- 但能作为同步失败后的兜底恢复入口

## 9. 会话同步策略

### 9.1 同步触发时机

- 用户在聊天页保存会话 skills 时
- Web 会话启动时，如存在待生效配置
- 手动点击重同步时

### 9.2 同步方式

采用“整目录重建”策略，不做增量 patch。

流程：

1. 读取会话当前绑定的 skills
2. 清空 `data/skills/sessions/<chatJid>/active/`
3. 将每个选中的全局 skill 目录完整复制到会话生效目录
4. 成功后更新同步状态为 `synced`
5. 任一步失败则记录为 `failed`

### 9.3 推荐原因

- 逻辑简单，可预测
- 不会残留旧 skill 文件
- 避免复杂增量同步带来的遗漏与脏状态

### 9.4 同步失败语义

- 绑定关系保存成功后，即使同步失败，也不回滚用户选择
- 前端应展示“已保存，但同步失败”
- 用户可再次保存或手动触发重同步

## 10. 前端交互设计

### 10.1 管理员页：`/skills`

在现有侧边栏中新增仅 `admin` 可见入口，与“用户管理”同级。

页面结构建议分为三部分：

- 顶部操作栏
  - 新建 skill
  - 上传 zip
  - 刷新
- 左侧目录树
  - 展示全局库目录结构
  - 支持展开、折叠、选择节点
  - 提供新建、重命名、删除、移动操作入口
- 右侧内容区
  - 目录节点：显示目录内容与基础信息
  - 文本文件：显示编辑器，可保存
  - 二进制文件：显示不可在线编辑提示

交互要求：

- 文件修改未保存时，切换节点前需要确认
- zip 上传成功后，自动刷新目录树并定位到新 skill
- 删除目录前必须确认

### 10.2 用户只读页：`/skills/catalog`

该页面允许 `user` 和 `admin` 访问。

用途：

- 浏览全局 skill 列表
- 查看 skill 的说明

展示建议：

- 列表或卡片形式展示全局 skills
- 优先预览 `SKILL.md`
- 仅只读，不出现编辑按钮

### 10.3 聊天页会话绑定入口

在聊天页新增 “Skills” 按钮、会话设置按钮或等价入口。

打开后展示抽屉或弹窗：

- 展示全局 skill 列表
- 支持多选
- 支持查看选中 skill 的 `SKILL.md` 说明
- 显示“当前会话已选择 N 个 skills”

保存后反馈以下之一：

- `已保存，等待会话启动时生效`
- `已保存，并已同步到当前运行会话`
- `已保存，但同步失败`

### 10.4 新建会话时的初始选择

由于需求明确要求“会话创建时可选”，因此新建 Web 会话流程也需要支持初始 skills 选择。

建议交互：

- 用户点击“新建会话”时，在创建流程中提供可选的 skills 多选区域
- 用户可直接以“无 skills”创建，也可带着已选 skills 创建
- 创建成功后，聊天页继续通过会话设置入口支持后续修改

### 10.5 用户侧范围收敛

普通用户侧不提供完整文件树浏览。

原因：

- 需求仅要求查看全局 skill 列表并进行会话选择
- 保持权限模型简单
- 避免把管理端复杂文件操作暴露到用户侧

## 11. 权限与安全

### 11.1 权限边界

- 管理员可以管理全局库
- 普通用户只能只读查看全局库
- 普通用户只能修改自己有权访问的 `web:*` 会话绑定

### 11.2 路径安全

所有文件系统操作都必须执行：

- 路径规范化
- 根目录 containment 校验
- 拒绝越权访问全局 skills 根目录之外的任何路径

### 11.3 文件类型安全

- 文本与二进制文件应区分处理
- 编辑接口不应盲目返回二进制内容

### 11.4 zip 安全

- 解压必须在临时目录完成
- 校验通过后再移动到目标目录
- 拒绝目录穿越型压缩包内容

## 12. 错误处理

### 12.1 管理员场景

需要明确区分以下错误：

- 目录树加载失败
- 文件读取失败
- 文件保存失败
- 重命名冲突
- 删除失败
- zip 同名冲突
- zip 缺少 `SKILL.md`
- zip 结构非法

### 12.2 用户场景

- 会话 skills 加载失败不应导致聊天主页面崩溃
- 保存绑定成功但同步失败时，需明确提示
- 技能目录只读接口失败时，应允许用户重试

## 13. 测试策略

### 13.1 后端测试

建议覆盖：

- 目录扫描与 skill 列表生成
- 路径 containment 校验
- 文本文件读写
- 文件/目录的新建、删除、移动、重命名
- zip 解压校验、同名冲突、结构非法、缺少 `SKILL.md`
- 会话绑定 CRUD
- 会话运行中与未运行时的同步分支
- 同步失败状态记录
- 权限校验

### 13.2 前端测试

建议覆盖：

- 管理员页目录树加载与节点切换
- 文本文件编辑与保存
- 未保存离开确认
- zip 上传成功与失败提示
- 用户只读 skill 列表展示
- 聊天页会话 skills 选择、保存与状态回显

### 13.3 集成测试

至少覆盖两条主链路：

1. 管理员上传 zip，成功生成新的全局 skill
2. 普通用户为某个 Web 会话选择 skill，保存后完成同步或进入待同步状态

## 14. 实施边界与后续扩展

### 14.1 本期边界

本期以可用、清晰、低风险为优先：

- 只支持 Web 会话
- 只做文件系统方案
- 不引入 skill 版本、历史记录、审核流

### 14.2 未来扩展方向

未来如有需要，可在此设计上扩展：

- 群组会话绑定 skills
- 全局库搜索与标签
- skill 版本与回滚
- 全局 skill 更新后提醒哪些会话需要重同步
- 只读文件树预览增强

## 15. 实现建议摘要

本设计建议采用“文件系统为真源、数据库只存绑定关系”的方式实现 skills 管理。

其核心原则是：

- 全局库与会话生效目录分离
- 管理员负责维护全局库
- 用户只读查看全局库，并为自己的 Web 会话选择 skills
- 保存绑定后按会话运行态决定立即同步或延迟同步

该方案能够在不重构现有底层 skill 机制的前提下，为项目新增一套清晰、可控、可测试的 skills 管理能力。
